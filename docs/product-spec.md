# AI Travel Offer Agent — Product Specification

> **Источник API:** официальная документация Tourvisor `tv-search-gateway 1.2.1`, проверена 15 июля 2026 года:  
> https://api.tourvisor.ru/search/docs  
>
> Все значения, не подтверждённые документацией или реальным ответом API, помечены как гипотезы первой беты.


## 1. Назначение пилота

AI Travel Offer Agent помогает одному турагенту превратить свободное сообщение клиента в структурированный поиск Tourvisor и готовую подборку туров.

Пилот ограничен одним турагентом, одним Telegram-ботом и одним доступом Tourvisor. В него не входят личный кабинет, PDF, биллинг, multi-tenant SaaS, CRM и сложная инфраструктура.

Основной pipeline:

```text
Сообщение клиента
→ LLM Parser
→ TravelSearchRequest
→ Validator
→ одно объединённое уточнение при необходимости
→ TourvisorSearchParams[]
→ параллельные поиски по странам
→ NormalizedTour[]
→ жёсткая фильтрация
→ ranking
→ top-3 по каждой стране
→ TravelOffer
```

## 2. Роли компонентов

### LLM Parser

LLM может:

- извлекать значения из свободного текста;
- нормализовать даты, состав туристов и предпочтения;
- классифицировать требования как обязательные или желательные;
- формировать один объединённый уточняющий вопрос;
- писать объяснение результата.

LLM не может:

- придумывать ID Tourvisor;
- придумывать цену, рейтинг, доступность и характеристики отеля;
- самостоятельно исключать туры;
- самостоятельно рассчитывать score.

### Детерминированный backend

Код отвечает за:

- валидацию JSON;
- разрешение ID по справочникам Tourvisor;
- создание поисковых параметров;
- polling;
- нормализацию ответа;
- фильтрацию;
- ranking;
- дедупликацию;
- дисклеймеры;
- безопасное логирование.

## 3. Основной runtime-сценарий

### 3.1. Начало

Турагент отправляет сообщение, содержащее запрос клиента.

Пример:

> Турция или Египет, вылет из Екатеринбурга, двое взрослых и ребёнок 8 лет, 7–9 ночей, до 250 тысяч, всё включено, отель от 4 звёзд.

Система создаёт `TravelSearchRequest`.

### 3.2. Проверка обязательных данных

Критически обязательны:

- город вылета;
- минимум одна страна;
- диапазон дат не более 21 дня;
- ночи от 1 до 28, диапазон не более 10;
- 1–6 взрослых;
- возраст каждого ребёнка, максимум трое детей 0–17 лет.

Бюджет технически не обязателен, но при его отсутствии система делает одну попытку уточнения.

### 3.3. Уточнение

Все недостающие данные объединяются в один вопрос.

Пример:

> Подскажите даты вылета, возраст ребёнка и хотя бы ориентировочный общий бюджет. Если бюджет пока не определён, напишите «без бюджета».

**Гипотеза первой беты:** разрешён один объединённый цикл уточнения. После ответа:

- критические поля заполнены → поиск разрешён;
- бюджет не указан или явно отклонён → поиск разрешён без `priceTo`;
- критические поля всё ещё отсутствуют → поиск не запускается, система возвращает список недостающих данных без повторного автоматического вопроса.

### 3.4. Несколько стран

Один поиск Tourvisor принимает один `countryId`. Поэтому для Турции и Египта создаются два независимых запроса, запускаемых параллельно.

Результаты выводятся отдельными блоками по странам. Внутри каждой страны выбирается до трёх разных отелей.

### 3.5. Успешный результат

Система:

1. получает результаты;
2. нормализует каждый тур;
3. применяет жёсткие фильтры;
4. рассчитывает score;
5. оставляет лучший тур каждого отеля;
6. выбирает top-3 по каждой стране;
7. формирует блок турагенту и готовый текст клиенту.

### 3.6. Нет результатов

Если поиск завершён и raw-выдача пуста:

- состояние страны: `empty`;
- турагенту сообщается, какие параметры можно ослабить;
- остальные страны продолжают обрабатываться.

Если raw-выдача была, но после фильтров кандидатов нет, система сообщает причины массового исключения: бюджет, питание, категория, даты и другие подтверждённые требования.

### 3.7. Ошибка одной страны

Ошибка одного направления не отменяет результаты другого.

Общий статус:

- есть хотя бы один пригодный результат → `completed_with_warnings`;
- нет результатов ни по одной стране → `failed`.

### 3.8. Ошибка внешнего API

Фатальные ошибки:

- 401/403;
- невалидные параметры;
- неизвестный `searchId`.

Временные ошибки:

- 429;
- 5xx;
- сетевой timeout.

Для временных ошибок допускаются две повторные попытки с задержкой 2 и 4 секунды. Это гипотеза первой беты.

### 3.9. Невалидный ответ LLM

Если parser вернул невалидный JSON:

1. выполнить одну repair-попытку с исходным текстом и ошибками схемы;
2. повторно проверить;
3. при повторной ошибке не запускать Tourvisor;
4. вернуть турагенту нейтральное сообщение о невозможности разобрать запрос и попросить отправить его более структурированно.

LLM repair не может добавлять значения, которых нет в исходном сообщении.

## 4. Контроль доступности и актуальности

Поисковая цена и доступность не являются гарантированными.

Поля `hotelPlace` и `flightPlace` сохраняются как raw-коды, но не интерпретируются до получения официальной расшифровки.

Метод `/tours/{tourId}/flights` может использоваться для актуализации финальных вариантов и возвращает актуальную цену, `noPlaces` и `onDemand`. Такой вызов тарифицируется как поисковый запрос.

**Гипотеза первой беты:** актуализация финалистов является конфигурируемой функцией. Backend на mock-данных должен поддерживать её, но реальный default определяется после проверки стоимости и скорости на пилоте.

Даже после актуализации клиентский текст содержит дисклеймер о необходимости финального подтверждения цены и мест у туроператора.

## 5. Формат результата

### Блок турагенту

Содержит:

- как понят запрос;
- допущения;
- неизвестные или непроверяемые требования;
- статус поиска по странам;
- top-3;
- причины первого места;
- компромиссы;
- предупреждения по цене и местам;
- что перепроверить.

### Блок клиенту

Содержит:

- короткое вступление;
- до трёх вариантов по каждой стране;
- отель, курорт, категория и рейтинг при наличии;
- дата, ночи, питание, номер при наличии;
- цена;
- один объективный аргумент выбора;
- один краткий компромисс;
- дисклеймер.

Клиентский текст не содержит внутренний score, служебные коды, технические ошибки и неподтверждённые маркетинговые утверждения.

## 6. Критерии готовности backend на mock

Backend готов к интеграции Tourvisor, если он умеет:

- валидировать `TravelSearchRequest`;
- создавать отдельные поиски по странам;
- обрабатывать lifecycle;
- нормализовать fixtures;
- применять фильтры;
- рассчитывать ranking;
- выбирать top-3 разных отелей;
- формировать `TravelOffer`;
- обрабатывать partial, empty, error и invalid JSON;
- не использовать непроверенные поля как факты.

## 7. Метрики пилота

Минимально логируются:

- время до первого результата;
- полное время workflow;
- число уточнений;
- число raw-туров;
- причины исключения;
- число кандидатов;
- ручные правки турагента;
- отправлен ли вариант клиенту;
- стоимость LLM;
- число тарифицируемых Tourvisor-вызовов.
